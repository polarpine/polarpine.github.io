    SQL Injection

    By Polarpine August 10, 2014 Tweet Like +1

    To explain what SQL Injection Issues are, we need to take a step back and look at what SQL is and what it can do. SQL or Structured Query Language is a programming language designed to access and manipulate databases. SQL views the entire world through the lens of a table. Everything it organizes, adds, removes, queries, is in the form of a table. It sounds pretty simple yet it had powerful implications because SQL is so widely used.

    Let’s look at a simple example fo how SQL would access a database. Imagine you’re in a monthly book club. This book club has a database associated with it ‘cause one of your members just happens to be an overzealous code-monkey. Stored in your book club database are:

    • book club members
    • books you’ve read
    • authors
    • books you want to read
    • who’s hosting the meeting
    • how many times Sharon’s brought that damn tuna casserole
    • whether a book has been made into a movie (transfer to movie club!)

    Now, you want to access the database to see when you’re supposed to be hosting this rotating shindig. Conveniently, your valued code-monkey set up an online interface where you can put your name into a field labeled ‘when am I hosting?’ and it returns a lists of dates.

    SQL is the go-between in this instance that takes what you fed in, your name, and returned a value based on pre-set values. In SQL you just asked something like this:

    SELECT host_date
FROM book_club
WHERE book_club_member = 'Sharon';

    In the above code, SQL searches from the TABLE ‘book_club’ and returns values WHERE ‘book_club_member’ equals ‘Sharon’. Meaning, if and when your name matches a date in the ‘host_date’ column it will return or SELECT that value for you.

    That’s all fine and good though see that tricky little semi-colon? It signals to SQL that the command is closed. What if instead of passing in your name, Sharon, into the database, you decided to put foo';DROP TABLE book_club? Note the added semi-colon into the entry. SQL reads your query as:

    SELECT host_date
FROM book_club
WHERE book_club_member = 'foo'; DROP TABLE book_club;'

    Or put more explicitly

    SELECT host_date
FROM book_club
WHERE book_club_member = 'foo'; 

DROP TABLE book_club;'

    SQL is holding two commands, first SELECT items that match your query in TABLE ‘book_club’, second DROP TABLE ‘book_club’. Essentially what this will do is find a null value for your query request (unless foo is in your book club), then move on to the next command DROP TABLE. Deleting the table containing all the data for ‘book_club’. Goodbye Sharon’s casserole count. Hello angry code-monkey.

    Losing your book club information may not be too dire, losing all your banking information would be. If a database accepting user input is not sanitizing this input, removing malicious characters, it could be vulnerable to SQL injection hacking.

    There are a few simple things coders can do to combat SQL Injections. First would be limiting access to databases the the bare minimum required to perform the tasks. Users unlikely need the ability to update or delete tables. Take this away. Second would be managing how data is stored. Not keeping all data in one big pool. Segregate data into smaller functional groups. The third, and perhaps most important, sanitize user input. There are a couple of different ways to to this but ultimately it limits what kind of characters or sets of characters can be passed into SQL. This can be done by escaping certain characters, setting parameterized querries (creating a go-between from the user to the database), and pattern matching.

    SQL Injections present a very real threat if not properly mitigated. It’s important that time and thought go into database management and access reducing the risk of SQL Injections and limiting vulnerabilities.
